# Tasks

Статус: рабочая очередь реализации Phuket Go

Текущий фокус: довести публичный сайт `website` до локального 10/10 и только потом возвращаться к релизной подготовке. До этого не деплоим, не загружаем облачные ресурсы и не начинаем publication workflow.

## Правила

- Этот файл используется как инженерная очередь задач для реализации.
- `done` ставим только для реально закрытых и проверенных этапов.
- `in_progress` только у одной задачи одновременно.
- Трекер `project-forum/phuket-go-task-map.md` не заменяет этот файл и не используется как источник требований.

## T1. База разработки и правила проекта

- Цель: зафиксировать обязательную основу перед активной разработкой.
- Источник: `docs/requirements.md`, `docs/00-project-control/project-instructions-for-codex.md`
- Объем: документация и базовая проверка стартовых условий.
- Роли/права: все роли косвенно.
- Критерий готовности: есть `docs/requirements.md`, есть `tasks.md`, подтверждены обязательные стартовые документы и техническая база.
- Тесты: проверка существования файлов, чтение ключевых документов, проверка локальной основы.
- Статус: `done`

## T2. Публичный сайт MVP: главная, каталог, карточка экскурсии

- Цель: собрать рабочий публичный сайт на `website` с SEO-страницами и карточками экскурсий.
- Источник: `docs/requirements.md`
- Объем: главная, каталог, карточки, статьи, FAQ, контакты, условия.
- Роли/права: гость, авторизованный пользователь.
- Критерий готовности: пользователь может открыть главную, каталог и отдельную экскурсию; данные берутся из рабочей базы; есть CTA на заявку; desktop/mobile верстка доведена до 10/10.
- Тесты: `website` build/typecheck, ручная проверка основных страниц, Playwright screenshots/visual audit ключевых страниц, базовая проверка локального рендера.
- Сделано: сайт читает рабочую базу 54 экскурсий, использует локальный media sync, строится без localhost media URL, главная и страница экскурсии получили первый polish.
- Осталось: финальный visual/content pass главной, каталога, карточек, страниц экскурсий, статей, FAQ, контактов и условий.
- Статус: `in_progress`

## T3. Форма заявки и сохранение лида

- Цель: дать пользователю возможность оставить заявку без оплаты и без обязательного логина.
- Источник: `docs/requirements.md`, `docs/05-lead-processing/lead-flow.md`
- Объем: форма на карточке, API сохранения лида, подтверждение после отправки.
- Роли/права: гость, авторизованный пользователь.
- Критерий готовности: заявка с обязательными полями создается, сохраняется и возвращает понятный результат пользователю.
- Тесты: backend unit/integration, contract tests, web form flow.
- Статус: `done`

## T4. Маршрутизация заявки и Telegram-уведомления

- Цель: автоматически отправлять заявку владельцу проекта и исполнителю.
- Источник: `docs/requirements.md`, `docs/05-lead-processing/lead-flow.md`, `docs/09-platforms-and-architecture/telegram-bots-map.md`
- Объем: Telegram routing, отдельная обработка дайвинга, статусы партнера.
- Роли/права: админ, партнер.
- Критерий готовности: новая заявка уходит в нужные Telegram-каналы, действия партнера меняют статус.
- Тесты: backend integration tests, test stubs/mocks для Telegram, status history checks.
- Статус: `done`

- Статус: `done`
