# Containerization Plan

Статус: [~] draft

## Цель

Зафиксировать, как Phuket Go будет запускаться локально и как контейнеризация будет развиваться дальше.

## Главное решение

За основу берем подход шаблона `vibe`.

Локально сначала используем Docker Compose для инфраструктуры, а не для всего приложения.

Простыми словами:

- база Postgres запускается в Docker;
- сайт, админка и backend на этапе разработки могут запускаться обычными командами проекта;
- позже backend можно упаковать в контейнер для сервера;
- публичный сайт и webapp при деплое могут быть статическими сайтами, если это соответствует выбранному хостингу.

## Что контейнеризуем в MVP

| Часть | MVP | Как |
| --- | --- | --- |
| Postgres | да | `docker-compose.yml` |
| Тестовая Postgres | да | `docker-compose.yml` |
| Backend/API | позже | Dockerfile шаблона или подход хостинга |
| Website | нет как отдельный контейнер | сборка Astro, потом публикация |
| Webapp | нет как отдельный контейнер | сборка Vite, потом публикация |
| Telegram-боты | вместе с backend | отдельный модуль backend |
| Google Sheets | внешний сервис | без контейнера |
| Фото | локально на этапе подготовки | позже внешнее хранилище |

## Локальный запуск

Порядок для первой локальной версии:

1. Установить зависимости проекта.
2. Запустить Docker.
3. Поднять Postgres через Docker Compose.
4. Запустить backend.
5. Запустить website.
6. Запустить webapp.
7. Проверить заявку, Telegram-уведомления и запись данных.

## Что важно для не-программиста

Docker здесь нужен как коробка для базы данных.

Он позволяет не устанавливать Postgres вручную на компьютер.

Если проект переносится на другой компьютер, база поднимается одинаково.

## Текущее состояние

В шаблоне уже есть:

- `docker-compose.yml`;
- сервис `postgres`;
- сервис `postgres_test`;
- Postgres 18;
- отдельные порты для рабочей и тестовой базы.

## Что не делаем сейчас

- не контейнеризуем весь проект до старта разработки;
- не настраиваем production-деплой;
- не выбираем финальный cloud-сервис до этапа публикации;
- не переносим фото в Cloudinary/Yandex Object Storage сейчас;
- не усложняем архитектуру микросервисами.

## Будущее

Перед публикацией нужно выбрать финальную схему:

- Yandex Cloud;
- DigitalOcean из шаблона `vibe`;
- другой вариант, если будет причина.

Если выбираем Yandex Cloud, нужно отдельно описать:

- где живет backend;
- где живет Postgres;
- где хранятся фото;
- где публикуются `website` и `webapp`;
- как идут переменные окружения и секреты.

- [x] Понятно, что backend можно контейнеризовать позже.
- [x] Понятно, что website и webapp не обязаны быть контейнерами.
- [x] Понятно, что финальный cloud выбирается перед публикацией.
