# Architecture Overview

Статус: [~] draft

## Базовое архитектурное правило

За основу берем решения GitHub-шаблона `vibe`.

Если ранние решения Phuket Go конфликтуют с решениями шаблона, приоритет у шаблона.

Это значит:

- публичные SEO-страницы делаем в `website`;
- личный кабинет и админку делаем в `webapp`;
- backend/API делаем в `backend`;
- общие схемы данных держим в `packages/contracts`;
- мобильную часть не трогаем до отдельного решения.

Продуктовую логику Phuket Go не отменяем, а адаптируем под структуру шаблона.

## Клиенты

Порядок развития:

1. Сайт.
2. Telegram Mini App.
3. Android.
4. iPhone.

MVP начинается с сайта.

Telegram Mini App идет после сайта.

Android и iPhone - позже.

## Backend

Нужен единый backend, который в будущем смогут использовать:

- сайт;
- Telegram Mini App;
- Android;
- iPhone.

Backend должен поддерживать:

- пользователей;
- авторизацию;
- экскурсии;
- заявки;
- партнеров/исполнителей;
- отзывы;
- статьи;
- баллы и уровни;
- админку;
- Telegram-ботов.

## Database

Временный этап: Google Sheets.

Целевая база: Postgres.

Архитектура должна учитывать будущий перенос данных в Postgres.

## Admin

Админка нужна в MVP.

В MVP админ один: владелец проекта.

Админка управляет:

- экскурсиями;
- заявками;
- пользователями;
- отзывами;
- статьями;
- партнерами/исполнителями;
- комиссиями;
- аналитикой.

## Partner Portal

Партнерский кабинет в MVP не нужен.

Партнер работает через Telegram-бот.

Но архитектура должна учитывать будущий партнерский доступ:

- партнер сможет зайти в свой раздел;
- партнер сможет дополнять свои экскурсии;
- партнер сможет предлагать изменения описания, фото, программы, вариантов и тарифов;
- изменения партнера должны проходить модерацию администратора перед публикацией.

## Telegram-боты

В MVP нужны:

- бот для партнеров;
- бот для пользователей.

Бот для партнеров:

- получает заявку;
- дает кнопки "Взята в работу", "Отклонено", "Оказана".

Бот для пользователей:

- показывает статус заявки;
- помогает смотреть историю заявки.

## Мультиязычность

MVP только на русском.

Но архитектура должна учитывать будущие языки.

## Аналитика и логирование

В MVP нужны аналитика и логирование.

Нужно фиксировать:

- заявки;
- смену статусов;
- действия администратора;
- действия партнеров в Telegram-боте;
- ошибки маршрутизации;
- ошибки отправки уведомлений.

## Безопасность

Чувствительные данные:

- телефон;
- Telegram;
- email.

Админка должна быть защищена двухфакторной авторизацией.

## Резервные копии

Для Google Sheets нужны резервные копии.
