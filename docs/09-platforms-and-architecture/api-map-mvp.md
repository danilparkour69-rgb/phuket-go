# API Map MVP

Статус: [x] done

Кодинг проекта пока не начинаем.

Документ описывает API-карту MVP Phuket Go: какие действия нужны публичному сайту, личному кабинету, админке, Telegram-ботам и backend.

## Главное правило

За основу берем решения GitHub-шаблона `vibe`.

API должно быть единым для:

- `website` - публичный сайт и SEO-страницы;
- `webapp` - личный кабинет и админка;
- `backend` - бизнес-логика;
- `packages/contracts` - общие схемы запросов и ответов;
- будущих клиентов: Telegram Mini App, Android, iPhone.

MVP остается Phuket Go, но API проектируем с заделом на:

- Bangkok Go;
- Pattaya Go;
- Thailand Go;
- Vietnam Go;
- Asia Go;
- World Go.

## Типы API

| Тип | Для чего нужен | Доступ |
| --- | --- | --- |
| Public API | Каталог, экскурсии, статьи, отзывы, форма заявки | Публичный |
| Account API | Профиль, история заявок, баллы | Только авторизованный пользователь |
| Admin API | Управление заявками, экскурсиями, статьями, отзывами, комиссиями | Только админ |
| Partner Bot API | Обработка заявок партнером через Telegram | Только Telegram-бот партнера |
| User Bot API | Просмотр статуса заявки через Telegram | Только Telegram-бот пользователя |
| Internal API | Google Sheets, уведомления, расчет комиссий, аудит | Только backend |

## Public API

Публичный сайт использует `website`.

### Каталог экскурсий

| Действие | Назначение | Доступ |
| --- | --- | --- |
| Получить список экскурсий | Показать каталог | Public |
| Получить фильтры каталога | Категории и диапазоны цен | Public |
| Получить карточку экскурсии | Показать страницу экскурсии | Public |
| Получить похожие экскурсии | Блок рекомендаций | Public |
| Получить отзывы экскурсии | Показывать только опубликованные отзывы | Public |

Нужно учитывать:

- город по умолчанию: Phuket;
- страна по умолчанию: Thailand;
- язык по умолчанию: русский;
- публичная цена: RUB;
- внутренняя цена: THB.

### Статьи

| Действие | Назначение | Доступ |
| --- | --- | --- |
| Получить список статей | Страница `/articles` | Public |
| Получить статью | Страница `/articles/{slug}` | Public |
| Получить экскурсии к статье | Связать статью с каталогом | Public |

### FAQ, контакты, условия

| Действие | Назначение | Доступ |
| --- | --- | --- |
| Получить FAQ | Частые вопросы | Public |
| Получить условия бронирования | Правила заявки, оплаты, отмены | Public |
| Получить контакты | Способы связи | Public |

### Создание заявки

| Действие | Назначение | Доступ |
| --- | --- | --- |
| Создать заявку | Гость или пользователь отправляет форму | Public |

Обязательные поля:

- имя;
- телефон;
- `excursion_id`.

Дополнительные поля:

- Telegram;
- дата;
- количество людей;
- комментарий;
- источник страницы.

После создания заявки backend должен:

1. создать заявку со статусом `new`;
2. сохранить заявку в Google Sheets;
3. показать заявку в админке;
4. отправить заявку админу в Telegram;
5. отправить заявку партнеру в Telegram;
6. если пользователь авторизован, связать заявку с его профилем.

## Partner Bot API

Использует backend и Telegram-бот партнера.

### Получение заявки

Партнер получает в Telegram:

- номер заявки;
- название экскурсии;
- имя клиента;
- телефон;
- Telegram клиента, если есть;
- дату;
- количество людей;
- комментарий;
- кнопки действий.

### Действия партнера

| Действие | Назначение | Доступ |
| --- | --- | --- |
| Взять заявку в работу | Статус `accepted` | Partner Bot |
| Отклонить заявку | Статус `declined` | Partner Bot |
| Отметить оказанной | Статус `completed` | Partner Bot |

После каждого действия backend должен:

1. обновить статус заявки;
2. записать историю статуса;
3. уведомить админа;
4. уведомить пользователя, если есть связь;
5. при `completed` создать или обновить комиссию.

## User Bot API

Использует backend и Telegram-бот пользователя.

| Действие | Назначение | Доступ |
| --- | --- | --- |
| Связать Telegram с пользователем | Чтобы показывать заявки | User Bot |
| Получить мои заявки | Список заявок | User Bot |
| Получить статус заявки | Текущий статус | User Bot |
| Получить уведомление | Изменение статуса | User Bot |

Если пользователь не авторизован, связь возможна по телефону или Telegram, но это нужно реализовывать аккуратно.

## Internal API

Только backend.

| Действие | Назначение |
| --- | --- |
| Записать заявку в Google Sheets | Временное хранилище заявок |
| Обновить строку Google Sheets | Смена статуса или комиссии |
| Отправить Telegram-уведомление админу | Контроль заявки |
| Отправить Telegram-уведомление партнеру | Передача заявки |
| Отправить Telegram-уведомление пользователю | Статус заявки |
| Рассчитать комиссию | После оказания услуги |
| Начислить баллы | Регистрация, экскурсия, отзыв |
| Записать audit log | Важные действия |

## Что должно быть в `packages/contracts`

Когда начнется разработка, в общие схемы нужно вынести:

- `LeadCreateInput`;
- `LeadStatus`;
- `LeadDto`;
- `ExcursionCardDto`;
- `ExcursionPageDto`;
- `ArticleDto`;
- `ReviewDto`;
- `UserProfileDto`;
- `PointsDto`;
- `PartnerDto`;
- `CommissionDto`;
- `AdminLeadDto`;
- `ApiError`.

Цель: `website`, `webapp`, `backend` и будущие приложения используют одинаковые форматы данных.

## Защита доступа

| Зона | Кто имеет доступ |
| --- | --- |
| Public API | Все |
| Account API | Только авторизованный пользователь |
| Admin API | Только админ |
| Partner Bot API | Только проверенный Telegram-бот партнера |
| User Bot API | Пользователь через Telegram |
| Internal API | Только backend |

## Что не делаем в MVP API

- онлайн-оплату;
- самостоятельное создание карточек партнером;
- удаление карточек партнером;
- полноценный партнерский кабинет;
- Telegram Mini App;
- мобильное приложение;
- мультиязычность в интерфейсе;
- геолокационную игру;
- награды и подарки.

## Задел на масштабирование

API не должно быть жестко привязано только к Пхукету.

Даже если в MVP значения будут по умолчанию, позже API должно поддержать:

- `country`;
- `city`;
- `language`;
- `currency`;
- `service_category`;
- несколько партнеров в разных городах;
- единый аккаунт пользователя по разным городам и странам.
