# Data Structure MVP

Статус: [x] done

Кодинг проекта пока не начинаем.

Документ описывает структуру данных MVP Phuket Go с учетом шаблона `vibe`, Google Sheets и будущего Postgres.

## Главное правило

За основу берем решения GitHub-шаблона `vibe`.

Данные должны быть описаны так, чтобы их могли использовать:

- `website` - публичный сайт и SEO-страницы;
- `webapp` - личный кабинет и админка;
- `backend` - заявки, Telegram-боты, Google Sheets, бизнес-логика;
- `packages/contracts` - общие правила данных и форматы.

Если ранние решения Phuket Go конфликтуют со структурой `vibe`, адаптируем решения под `vibe`.

## Задел на масштабирование

MVP запускается как Phuket Go, но структура данных должна учитывать будущий путь:

1. Phuket Go.
2. Bangkok Go.
3. Pattaya Go.
4. Thailand Go.
5. Vietnam Go.
6. Asia Go.
7. World Go.

Поэтому в будущей Postgres-структуре нужно предусмотреть сущности:

- `Country`;
- `Region`;
- `City`;
- `Location`;
- `Currency`;
- `Language`;
- `ServiceCategory`.

В MVP эти сущности можно держать упрощенно или значениями по умолчанию:

- страна: Thailand;
- город: Phuket;
- язык: ru;
- валюта партнера: THB;
- публичная валюта MVP: RUB.

Главное: не проектировать данные так, будто Phuket Go навсегда останется одним городом и одной категорией услуг.

## Этапы хранения данных

### Этап 1. Документы

Сейчас данные экскурсий хранятся в документах:

- `docs/03-service-catalog/excursions/`
- `docs/03-service-catalog/excursions/README.md`

Это база для подготовки контента.

### Этап 2. Google Sheets

В MVP Google Sheets временно хранит заявки.

Одна строка = одна заявка.

Google Sheets не должен становиться полноценной базой всего проекта.

### Этап 3. Postgres

После подготовки MVP данные переносятся в Postgres.

Postgres станет основной базой для:

- пользователей;
- экскурсий;
- заявок;
- статусов;
- партнеров;
- отзывов;
- статей;
- баллов;
- комиссий.

## Миграция в Postgres

Когда переходим с Google Sheets на Postgres:

1. Создаем таблицы по сущностям из этого документа.
2. Переносим заявки из Google Sheets в `Lead`.
3. Переносим ключевые даты статусов в `LeadStatusHistory`.
4. Создаем `Commission` по завершенным заявкам.
5. Проверяем связи с пользователями, партнерами и экскурсиями.
6. После проверки Google Sheets остается резервным экспортом или отключается.

## Правило для будущего партнерского кабинета

Партнер не может удалить карточку экскурсии.

Партнер может:

- скрыть карточку;
- предложить изменения;
- отправить запрос администратору на удаление.

Администратор решает, удалять карточку полностью или нет.

Причина: удаление может сломать историю заявок, отзывы, SEO-страницу, аналитику и комиссии.

## Definition of Done этапа

Этап структуры данных считается готовым, если:

- описаны главные сущности MVP;
- указаны обязательные поля;
- отдельно описан Google Sheets;
- отдельно описан будущий Postgres;
- данные привязаны к `website`, `webapp`, `backend`, `packages/contracts`;
- учтены заявки, экскурсии, пользователи, партнеры, отзывы, статьи, баллы и комиссии.
