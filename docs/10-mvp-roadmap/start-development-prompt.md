## Промпт

```text
Продолжаем Phuket Go. Начинаем локальную разработку MVP.

Используй сохраненную инструкцию владельца проекта и актуальные документы.

Сначала не пиши фичи вслепую. Прочитай документы:

1. сохраненную инструкцию владельца проекта
2. docs/10-mvp-roadmap/development-handoff-map.md
3. docs/10-mvp-roadmap/immediate-pre-coding-actions.md
4. docs/10-mvp-roadmap/technical-requirements-mvp.md
5. docs/09-platforms-and-architecture/data-structure-mvp.md
6. docs/09-platforms-and-architecture/api-map-mvp.md
7. docs/10-mvp-roadmap/page-blocks-map-mvp.md
8. docs/03-service-catalog/excursions-site-data-mvp.json
9. docs/03-service-catalog/media/excursions/media-developer-handoff-map.md

Главные правила:

- Используем шаблон vibe.
- Если решение Phuket Go конфликтует с vibe, берем подход vibe и адаптируем продукт.
- Публичный сайт, каталог, страницы экскурсий, статьи, FAQ, контакты и условия делаем в website.
- Личный кабинет и админку делаем в webapp.
- Backend/API делаем в backend.
- Общие схемы данных делаем в packages/contracts.
- Код должен запускаться локально.
- Онлайн-оплату не делаем.
- SMS-авторизацию по телефону не делаем.
- Партнерский кабинет в MVP не делаем.
- Telegram Mini App и мобильные приложения не делаем.
- Фото берем из media-manifest.json и final/carousel внутри папки направления.
- На фото больше не зависаем: берем доступные фото, соответствующие карточке экскурсии.
- Финальную чистку фото, домен, хостинг и боевые Telegram-боты оставляем на этап перед публикацией.

Первый технический шаг:

1. Проверить структуру репозитория и текущий статус файлов.
2. Прочитать README/AGENTS/CLAUDE и документы шаблона vibe.
3. Проверить package scripts.
4. Составить короткий план локального запуска.
5. Только после этого начинать минимальную реализацию MVP по этапам.
```
