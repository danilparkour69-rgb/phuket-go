# Codex Development Brief

Статус: [x] done for planning

Этот документ нужен для будущего старта разработки. Сейчас кодинг проекта не начинается.

## Проект

Phuket Go.

Миссия: Пхукет доступен в одно касание.

## Главная гипотеза MVP

Русскоязычные туристы будут оставлять заявки на экскурсии через Phuket Go, если карточки оформлены удобно, есть доверие и связь через Telegram.

## Что входит в MVP

- сайт;
- каталог экскурсий;
- отдельная страница каждой экскурсии;
- статьи;
- FAQ;
- контакты;
- условия бронирования;
- заявка без оплаты;
- Google Sheets как временное хранилище заявок;
- админка;
- Telegram-бот для партнеров;
- Telegram-бот для пользователей;
- регистрация;
- история заявок;
- баллы и первый уровень;
- отзывы после одобрения;
- аналитика заявок и комиссии.

## Что не входит в MVP

- онлайн-оплата;
- партнерский кабинет;
- Telegram Mini App;
- Android/iPhone;
- недвижимость;
- байки;
- визы;
- документы;
- полноценная игра;
- мультиязычность;
- самостоятельное добавление услуг партнерами.

## Стек

Используем шаблон `vibe`:

- публичный сайт: `website`, Astro;
- личный кабинет и админка: `webapp`, React + Vite;
- backend/API: `backend`, Bun + Hono;
- общие схемы данных: `packages/contracts`, Zod + TypeScript;
- язык: TypeScript;
- UI: Tailwind CSS и компоненты шаблона;
- авторизация: Telegram, Google, email; финально выбрать Firebase Authentication или встроенную авторизацию шаблона после технической проверки;
- фото: сначала локально в проектной папке, внешний сервис позже;
- временное хранение заявок: Google Sheets;
- целевая база: Postgres;
- хостинг после локальной проверки: Yandex Cloud;
- первая версия запускается локально на компьютере.

## Основные роли

- гость;
- авторизованный пользователь;
- партнер/исполнитель;
- администратор.

## Основные сценарии

- гость смотрит экскурсии, читает статьи, оставляет заявку без регистрации;
- пользователь видит историю заявок, баллы и первый уровень;
- партнер работает через Telegram-бот;
- админ управляет экскурсиями, заявками, пользователями, отзывами, статьями, комиссиями и аналитикой.

## Заявки

- обязательные поля: имя, телефон;
- дополнительные поля: Telegram, дата, количество людей, комментарий;
- заявка отправляется в Telegram-бот;
- заявка дублируется администратору;
- заявка уходит партнеру/исполнителю;
- партнер выбирает: "Взята в работу", "Отклонено", "Оказана";
- если партнер не отвечает, пользователь получает сообщение "Ожидаем подтверждения";
- пользователь видит статус в личном кабинете и через пользовательский Telegram-бот.

## Монетизация

- оплат на сайте нет;
- партнер платит обратную комиссию после оказания услуги;
- базовая комиссия по экскурсиям: 100 бат за человека;
- расчет комиссии раз в месяц;
- долги партнеров отмечаются вручную.

## Геймификация MVP

- регистрация = 100 баллов;
- 100 баллов = первый уровень;
- баллы нужны только для уровней;
- отзыв с фото дает больше баллов, чем текстовый отзыв;
- полноценная игра и награды позже.

## SEO и контент

- каждая экскурсия имеет отдельную страницу;
- есть общий каталог с фильтрами;
- отдельные страницы направлений в MVP не нужны;
- нужны статьи;
- нужны FAQ, полезные советы, условия бронирования, контакты;
- язык MVP: русский.

## Порядок разработки

1. Подготовка данных и структуры.
2. Сайт и SEO-страницы.
3. Админка.
4. Заявки и Google Sheets.
5. Telegram-боты.
6. Авторизация и личный кабинет.
7. Баллы и уровни.
8. Аналитика и комиссии.
9. Локальное тестирование.
10. Публикация на сервере.

## Свежие правила по фото

По фото больше не зависаем.

Для MVP берем доступные фото, которые уже лежат в папке конкретного направления и соответствуют карточке экскурсии.

Сайт должен брать фото через:

- `docs/03-service-catalog/media/excursions/{direction}/media-manifest.json`
- `docs/03-service-catalog/media/excursions/{direction}/final/carousel/`

Финальная чистка, замена слабых фото, проверка лиц и улучшение через ИИ переносятся на этап перед публикацией.

Главный документ:

- `docs/03-service-catalog/media/excursions/media-developer-handoff-map.md`
