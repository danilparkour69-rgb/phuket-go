## Что входит в MVP

### Публичный сайт

- главная страница;
- каталог экскурсий;
- фильтр по категории;
- фильтр по цене;
- отдельная SEO-страница каждой экскурсии;
- статьи;
- FAQ;
- условия бронирования;
- контакты;
- форма заявки.

### Заявка

Обязательные поля:

- имя;
- телефон.

Дополнительные поля:

- Telegram;
- дата;
- количество людей;
- комментарий.

После отправки:

- заявка сохраняется;
- заявка уходит админу в Telegram;
- заявка уходит партнеру в Telegram;
- пользователь видит подтверждение;
- авторизованный пользователь видит заявку в истории.

### Личный кабинет

- вход через Telegram, Google, email;
- профиль;
- история заявок;
- детали заявки;
- статусы;
- баллы;
- первый уровень;
- связь с Telegram-ботом.

### Админка

- заявки;
- экскурсии;
- пользователи;
- партнеры;
- отзывы;
- статьи;
- комиссии;
- аналитика.

### Telegram-боты

- бот партнера;
- бот пользователя;
- уведомления админу;
- смена статуса заявки партнером;
- просмотр статуса заявки пользователем.

## Что не входит в MVP

- онлайн-оплата на сайте;
- полноценный кабинет партнера;
- самостоятельное удаление карточек партнером;
- мобильные приложения;
- Telegram Mini App;
- полноценная игра;
- награды и подарки за баллы;
- англоязычная версия;
- недвижимость;
- визы;
- документы;
- байки;
- обмен денег;
- сложная CRM;
- автоматическая проверка оплаты комиссии;
- сложная система антинакрутки отзывов;
- телефонная авторизация через SMS.

## Что нужно учитывать из шаблона `vibe`

За основу берем решения GitHub-шаблона `vibe`.

Разделение:

| Часть | Где делать |
| --- | --- |
| Публичный сайт и SEO | `website` |
| Личный кабинет | `webapp` |
| Админка | `webapp` |
| Backend/API | `backend` |
| Общие схемы данных | `packages/contracts` |
| Мобильная версия | позже, через `mobile` branch |

Правило:

Если старое решение Phuket Go конфликтует с `vibe`, берем подход `vibe` и адаптируем продукт под него.

## Главные риски перед разработкой

| Риск | Что делать |
| --- | --- |
| Цены могут измениться к сезону | В карточках показывать рубли с предупреждением о курсе, THB хранить внутри |
| Не все условия экскурсий точные | Спорные поля оставлять как "уточнить при бронировании" |
| Фото пока готовим локально | Для MVP можно использовать разрешенные фото, сначала хранить локально, позже выбрать постоянное хранилище |
| Google Sheets временный | Не строить всю систему вокруг таблицы, позже перенос в Postgres |
| SMS-вход может быть платным | В MVP не использовать телефонную SMS-авторизацию |
| Партнер может не отвечать быстро | Показывать пользователю статус ожидания |
| Контент еще сырой | Перед запуском переписать тексты по правилу "продаем не экскурсии, а мечту, воспоминания и ощущения" |

## Что нужно уточнить до публикации онлайн

Эти пункты не мешают локальной разработке, но нужны перед реальным запуском:

- финальный хостинг;
- домен;
- точный сервис хранения фото для публикации;
- боевые Telegram-боты;
- доступ к Google Sheets;
- финальный список партнеров;
- актуальные цены перед сезоном;
- юридический текст условий бронирования;
- политика обработки персональных данных.

## Что нужно сделать до старта кодинга

Перед кодингом желательно закрыть контентный блок:

1. Финально почистить базу экскурсий.
2. Утвердить фотографии и обложки, убрать повторы.
3. Подготовить структуру главной страницы.
4. Подготовить тексты каталога.
5. Подготовить первые тексты страниц экскурсий.
6. Подготовить первые статьи.
7. Подготовить FAQ, условия бронирования и контакты.

## Решение по готовности

Технически проект готов к переходу в разработку после отдельной команды владельца проекта.

По плану следующий этап перед кодингом:

- контентная стратегия;
- тексты;
- SEO-материалы;
- финальная подготовка базы экскурсий.
