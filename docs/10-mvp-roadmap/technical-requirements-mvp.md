# Technical Requirements MVP

Статус: [x] done

Этот документ нужен для разработки Phuket Go.

## Цель MVP

Запустить локальную первую версию сервиса, где русскоязычный турист может:

- посмотреть экскурсии на Пхукете;
- прочитать полезные статьи;
- открыть отдельную страницу экскурсии;
- оставить заявку без оплаты на сайте;
- получить уведомление о заявке;
- видеть историю заявок и баллы после входа.

Администратор должен видеть заявки, экскурсии, пользователей, отзывы, статьи, комиссии и аналитику.

Партнер в MVP работает через Telegram-бота, а не через личный кабинет.

## Основа проекта

Используем шаблон:

- GitHub: `https://github.com/danilparkour69-rgb/phuket-go`
- исходный шаблон: `https://github.com/di-sukharev/vibe`

Важно: `vibe` - это стартовый шаблон, а не точная архитектура Phuket Go.

Если решения шаблона конфликтуют с ранними решениями Phuket Go, за основу берем решения `vibe`.

Продуктовые требования Phuket Go при этом сохраняем, но реализуем их способом, который лучше подходит структуре шаблона.

## Что должно быть в публичном сайте

### Главная

Итог: человек понимает, что Phuket Go помогает выбрать экскурсию на Пхукете без хаоса Telegram-постов.

Нужно показать:

- миссию: `Пхукет доступен в одно касание`;
- популярные экскурсии;
- категории;
- доверие: страховка, русскоязычные гиды, проверенные партнеры, отзывы;
- как работает заявка;
- статьи;
- FAQ;
- контакты.

### Каталог экскурсий

Итог: человек быстро выбирает подходящую экскурсию.

Нужно показать:

- карточки экскурсий;
- фильтр по категории;
- фильтр по цене;
- цену в батах крупно;
- расчетную цену в рублях мелко для русской версии;
- короткий эмоциональный текст;
- кнопку `Оставить заявку`.

Партнера в карточке не показываем.

### Страница экскурсии

Итог: человек понимает, какой день он получит, и оставляет заявку.

Нужно показать:

- название;
- фотогалерею;
- цену в батах крупно;
- расчетную цену в рублях мелко для русской версии;
- длительность;
- описание эмоции;
- программу и маршрут;
- что входит;
- что оплачивается отдельно;
- что взять с собой;
- ограничения;
- страховку;
- отзывы;
- FAQ по экскурсии;
- похожие экскурсии;
- связанные статьи;
- несколько кнопок `Оставить заявку` по странице.

Правило: продаем не экскурсии, а мечту, воспоминания и ощущения.

### Статьи

Итог: статьи приводят SEO-трафик и направляют человека к экскурсиям.

Нужно сделать раздел:

- `/articles`
- `/articles/{slug}`

В каждой статье должны быть:

- понятный ответ на вопрос;
- ссылки на подходящие экскурсии;
- переход в каталог.

## Что должно быть в заявке

Обязательные поля:

- имя;
- телефон.

Дополнительные поля:

- Telegram;
- дата;
- количество людей;
- комментарий.

После отправки:

- заявка сохраняется;
- заявка уходит администратору в Telegram;
- заявка уходит партнеру в Telegram;
- пользователь видит подтверждение;
- авторизованный пользователь видит заявку в истории.

Оплаты на сайте нет.

## Статусы заявки

Минимальные статусы MVP:

- `new` - новая заявка;
- `waiting_partner` - ожидаем ответ партнера;
- `accepted` - взята в работу;
- `declined` - отклонена;
- `completed` - услуга оказана;
- `cancelled` - отменена.

В истории нужно хранить:

- кто изменил статус;
- когда изменил статус;
- старый статус;
- новый статус.

## Telegram-бот для партнера

Итог: партнер получает заявку и быстро отвечает.

Партнер получает:

- название экскурсии;
- имя клиента;
- телефон;
- Telegram, если есть;
- дата;
- количество людей;
- комментарий;
- ссылка или номер заявки.

Кнопки:

- `Взята в работу`;
- `Отклонено`;
- `Оказана`.

Если партнер не отвечает, пользователю показываем статус ожидания.

## Telegram-бот для пользователя

Итог: пользователь может видеть статус заявки через Telegram.

Нужно:

- показать список заявок пользователя;
- показать статус каждой заявки;
- отправлять уведомления при смене статуса.

## Личный кабинет

Итог: авторизованный пользователь видит свою историю и баллы.

Нужно показать:

- профиль;
- историю заявок;
- статусы заявок;
- баллы;
- уровень.

Гость может оставлять заявку без регистрации.

## Авторизация MVP

Нужны способы входа:

- Telegram;
- Google;
- email.

Телефон через SMS в MVP не используем, потому что нужен бесплатный способ.

## Админка

Итог: владелец проекта может управлять MVP без программиста.

Админ видит:

- экскурсии;
- заявки;
- пользователей;
- партнеров;
- отзывы;
- статьи;
- комиссии;
- аналитику.

Админ может:

- создать/изменить экскурсию;
- скрыть экскурсию;
- посмотреть заявку;
- изменить статус заявки;
- одобрить или скрыть отзыв;
- создать/изменить статью;
- отметить комиссию как оплаченную;
- отметить долг партнера.

В MVP админ один.

## Отзывы

Итог: отзывы повышают доверие, но не публикуются без проверки.

Правила:

- отзыв публикует администратор вручную;
- звезд в MVP нет;
- текстовый отзыв обязателен;
- фото в отзыве возможно позже;
- отзыв можно оставить только после посещения экскурсии.

## Баллы и уровни

Итог: в MVP пользователь видит первый уровень и простые баллы.

Правила:

- регистрация = 100 баллов;
- 100 баллов = первый уровень;
- заказ экскурсии дает больше баллов, чем отзыв;
- отзыв с фото дает больше баллов, чем текстовый отзыв;
- реферал дает больше баллов, чем отзыв, но меньше, чем экскурсия.

Награды и подарки - позже.

## Комиссии

Итог: админ видит, сколько Phuket Go должен получить от партнера.

Правила MVP:

- оплата клиентом проходит у партнера;
- партнер платит Phuket Go после оказания услуги;
- базовая комиссия по экскурсиям: 100 THB за человека;
- расчет комиссии раз в месяц;
- долг партнера можно отметить вручную.

## Данные и хранение

В MVP:

- заявки временно сохраняются в Google Sheets;
- экскурсии подготовлены в документах;
- фото сначала храним локально в проектной папке;
- база проектируется с учетом будущего Postgres.

Позже:

- переносим данные в Postgres;
- партнерский кабинет добавляем после MVP;
- масштабируем на другие категории.

## Фото

Правила:

- фото из Telegra.ph источников можно использовать;
- оригиналов высокого качества пока нет;
- нужно заложить хранение фотогалереи для каждой экскурсии;
- нужно различать реальные фото, AI-улучшенные фото и AI-сгенерированные изображения;
- AI-изображения нельзя выдавать за реальные фото конкретной экскурсии;
- постоянное хранилище фото выберем позже.

## Ограничения по экскурсиям

Не продаем катание на слонах.

Если в исходной программе есть слоны, оставляем только форматы:

- кормление;
- купание/уход;
- наблюдение без катания.

## Цена

Публично показываем цену в батах крупно.

Для русской версии под ценой мелким текстом:

`примерно {price_rub} ₽ по текущему курсу`

Для будущей английской версии под ценой мелким текстом:

`about ${price_usd} at the current exchange rate`

Внутри базы и для партнера основная цена хранится в THB.

## Что не делаем в MVP

- онлайн-оплату;
- партнерский кабинет;
- самостоятельное добавление экскурсий партнерами;
- Telegram Mini App;
- мобильное приложение;
- английский язык;
- недвижимость;
- байки;
- визы;
- документы;
- полноценную игру;
- награды и подарки.

## Definition of Done для MVP

MVP можно считать готовым к локальной проверке, если:

- публичный сайт открывается локально;
- каталог экскурсий работает;
- отдельные страницы экскурсий работают;
- статьи, FAQ, контакты и условия бронирования есть;
- форма заявки отправляет данные;
- заявка сохраняется в Google Sheets;
- заявка приходит админу в Telegram;
- заявка приходит партнеру в Telegram;
- партнер может сменить статус заявки;
- пользователь получает уведомление о смене статуса;
- авторизованный пользователь видит историю заявок;
- админ видит заявки, экскурсии, отзывы, статьи, комиссии и аналитику;
- цена показывается в батах крупно и в рублях мелко для русской версии;
- слонов с катанием в продаже нет.
