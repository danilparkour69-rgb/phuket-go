# User Scenarios MVP

Статус: [x] done

Этот документ описывает, как должны работать основные роли Phuket Go в MVP.

## Архитектурная привязка

За основу берем решения GitHub-шаблона `vibe`.

Сценарии распределяются так:

- публичные страницы: `website`;
- личный кабинет: `webapp`;
- админка: `webapp`;
- заявки, статусы, Telegram-боты, Google Sheets: `backend`;
- общие правила данных: `packages/contracts`.

Если ранние решения Phuket Go конфликтуют со структурой `vibe`, сценарии адаптируем под `vibe`.

## Сценарий 1. Гость оставляет заявку без регистрации

### Цель

Гость должен быстро выбрать экскурсию и оставить заявку без обязательного входа.

### Шаги

1. Гость открывает сайт Phuket Go.
2. Видит главную страницу с обещанием `Пхукет доступен в одно касание`.
3. Переходит в каталог экскурсий.
4. Использует фильтры по категории и цене.
5. Открывает страницу экскурсии.
6. Смотрит фото, цену в рублях, маршрут, что входит, условия, отзывы.
7. Нажимает кнопку `Оставить заявку`.
8. Заполняет обязательные поля:
   - имя;
   - телефон.
9. Может заполнить дополнительные поля:
   - Telegram;
   - дата;
   - количество людей;
   - комментарий.
10. Отправляет заявку.
11. Видит сообщение, что заявка отправлена и менеджер свяжется с ним.
12. Дальше получает информацию через Telegram или телефон.

### Что происходит в системе

1. Заявка получает статус `new`.
2. Заявка сохраняется в Google Sheets.
3. Заявка отображается в админке.
4. Заявка отправляется администратору в Telegram.
5. Заявка отправляется партнеру в Telegram.
6. Если партнер не отвечает, статус становится `waiting_partner`.

### Итог

Заявка создана без регистрации.

Гость не видит историю заявок на сайте, но получает связь через мессенджер или телефон.

## Сценарий 2. Гость читает статью и переходит к экскурсии

### Цель

Статья должна привести человека к подходящей экскурсии.

### Шаги

1. Гость попадает на статью из поиска или с сайта.
2. Читает полезный материал.
3. Видит блок подходящих экскурсий.
4. Открывает страницу экскурсии.
5. Смотрит детали.
6. Оставляет заявку по сценарию гостя.

### Что происходит в системе

1. Заявка связывается с выбранной экскурсией.
2. Если возможно, фиксируется источник перехода: статья.

### Итог

Статья работает как SEO-вход и помогает продать идею поездки.

## Сценарий 3. Пользователь регистрируется

### Цель

Пользователь может войти и получить историю заявок, баллы и уровень.

### Шаги

1. Пользователь нажимает `Войти`.
2. Выбирает способ входа:
   - Telegram;
   - Google;
   - email.
3. Система создает профиль пользователя.
4. Пользователь получает 100 баллов.
5. Пользователь получает первый уровень.
6. Пользователь попадает в личный кабинет.

### Что происходит в системе

1. Создается пользователь.
2. Фиксируется способ входа.
3. Начисляется 100 баллов.
4. Создается запись в истории баллов.

### Итог

Пользователь авторизован и видит базовый личный кабинет.

## Сценарий 4. Авторизованный пользователь оставляет заявку

### Цель

Авторизованный пользователь должен оставить заявку быстрее, а потом видеть ее в истории.

### Шаги

1. Пользователь входит на сайт.
2. Открывает каталог или статью.
3. Переходит на страницу экскурсии.
4. Нажимает `Оставить заявку`.
5. Форма подставляет доступные данные пользователя, если они есть.
6. Пользователь заполняет недостающие поля.
7. Отправляет заявку.
8. Видит подтверждение.
9. Открывает личный кабинет.
10. Видит заявку в истории.

### Что происходит в системе

1. Заявка связывается с пользователем.
2. Заявка сохраняется в Google Sheets.
3. Заявка отображается в админке.
4. Заявка уходит админу и партнеру в Telegram.
5. Пользователь получает уведомление о создании заявки.

### Итог

Пользователь видит заявку в личном кабинете.

## Сценарий 5. Пользователь смотрит статус заявки

### Цель

Пользователь должен понимать, что происходит с заявкой.

### Вариант A. Через личный кабинет

1. Пользователь входит на сайт.
2. Открывает личный кабинет.
3. Видит список заявок.
4. Открывает нужную заявку.
5. Видит текущий статус.

### Вариант B. Через Telegram-бот

1. Пользователь открывает Telegram-бот Phuket Go.
2. Нажимает кнопку просмотра заявок.
3. Видит список своих заявок.
4. Выбирает нужную заявку.
5. Видит текущий статус.

### Статусы

- `new` - заявка отправлена;
- `waiting_partner` - ожидаем подтверждения;
- `accepted` - заявка взята в работу;
- `declined` - заявка отклонена;
- `completed` - экскурсия оказана;
- `cancelled` - заявка отменена.

### Итог

Пользователь не теряется после отправки заявки.

## Сценарий 6. Партнер получает заявку в Telegram

### Цель

Партнер должен быстро принять или отклонить заявку.

### Шаги

1. Партнер получает сообщение в Telegram-боте.
2. В сообщении видит:
   - номер заявки;
   - название экскурсии;
   - имя клиента;
   - телефон;
   - Telegram, если есть;
   - дату;
   - количество людей;
   - комментарий.
3. Партнер нажимает одну из кнопок:
   - `Взята в работу`;
   - `Отклонено`.
4. Если заявка взята в работу, партнер связывается с клиентом.
5. После оказания услуги партнер нажимает `Оказана`.

### Что происходит в системе

1. Статус заявки обновляется.
2. Изменение статуса сохраняется в истории.
3. Админ видит новый статус.
4. Пользователь получает уведомление.
5. Если статус `completed`, появляется комиссия к расчету.

### Итог

Партнер обработал заявку без входа в админку.

## Сценарий 7. Партнер отклоняет заявку

### Цель

Если партнер не может оказать услугу, заявка не должна зависнуть.

### Шаги

1. Партнер получает заявку.
2. Нажимает `Отклонено`.
3. Система меняет статус на `declined`.
4. Админ получает уведомление.
5. Пользователь получает уведомление.

### Что решает админ

Админ может:

- связаться с клиентом;
- предложить другую экскурсию;
- перенаправить заявку другому партнеру;
- закрыть заявку.

### Итог

Отклоненная заявка остается видимой админу.

## Сценарий 8. Админ обрабатывает заявки

### Цель

Админ должен видеть все заявки и контролировать процесс.

### Шаги

1. Админ входит в админку.
2. Открывает раздел заявок.
3. Видит список заявок и статусы.
4. Открывает конкретную заявку.
5. Видит:
   - клиента;
   - экскурсию;
   - дату;
   - количество людей;
   - партнера;
   - статус;
   - историю статусов;
   - расчет комиссии.
6. При необходимости меняет статус вручную.
7. При необходимости связывается с клиентом или партнером.

### Итог

Админ контролирует все заявки из одного места.

## Сценарий 9. Админ управляет экскурсиями

### Цель

Админ должен управлять каталогом без программиста.

### Шаги

1. Админ входит в админку.
2. Открывает раздел экскурсий.
3. Видит список экскурсий.
4. Может создать новую экскурсию.
5. Может изменить существующую экскурсию.
6. Может скрыть экскурсию с сайта.
7. Может добавить или заменить фото.
8. Может изменить цену, описание, маршрут, условия.

### Важные правила

- публичная цена показывается в рублях;
- внутренняя цена хранится в THB;
- под ценой показывается предупреждение о курсе;
- катание на слонах не продаем.

### Итог

Каталог можно обновлять через админку.

## Сценарий 10. Админ управляет отзывами

### Цель

Отзывы должны повышать доверие, но не публиковаться автоматически.

### Шаги

1. Админ открывает раздел отзывов.
2. Видит новые отзывы.
3. Проверяет, что клиент действительно был на экскурсии.
4. Одобряет отзыв или скрывает его.
5. Отзыв появляется в конце страницы экскурсии.

### Правила

- звезды в MVP не используем;
- текстовый отзыв обязателен;
- отзыв с фото в будущем дает больше баллов;
- отзыв можно оставить только после посещения экскурсии.

### Итог

На сайте публикуются только проверенные отзывы.

## Сценарий 11. Админ управляет статьями

### Цель

Статьи должны приводить SEO-трафик и вести к экскурсиям.

### Шаги

1. Админ открывает раздел статей.
2. Создает статью.
3. Добавляет заголовок, текст, SEO-данные.
4. Привязывает подходящие экскурсии.
5. Публикует статью.

### Итог

Статья появляется на сайте и помогает пользователю выбрать экскурсию.

## Сценарий 12. Админ смотрит комиссии и аналитику

### Цель

Админ должен понимать, сколько заявок, какие экскурсии работают и сколько денег должен партнер.

### Шаги

1. Админ открывает аналитику.
2. Видит:
   - количество заявок;
   - заявки по экскурсиям;
   - заявки по партнерам;
   - оказанные заявки;
   - сумму комиссии.
3. Открывает комиссии.
4. Видит начисления по партнерам.
5. Отмечает комиссию как оплаченную.
6. Если партнер не оплатил, ставит пометку долга.

### Правила

- базовая комиссия по экскурсиям: 100 THB за человека;
- партнер платит раз в месяц;
- комиссия появляется после статуса `completed`.

### Итог

Монетизация MVP контролируется в админке.

## Сценарий 13. Будущий партнерский доступ

### Цель

В MVP не реализуем, но архитектурно учитываем.

### Будущий сценарий

1. Партнер входит в свой раздел.
2. Видит только свои экскурсии.
3. Предлагает изменения:
   - описание;
   - фото;
   - программу;
   - условия;
   - варианты;
   - тарифы.
4. Может скрыть карточку со своего раздела, если экскурсия временно недоступна.
5. Не может удалить карточку экскурсии.
6. Если карточку нужно удалить полностью, партнер отправляет запрос администратору.
7. Изменения уходят админу на модерацию.
8. Админ одобряет или отклоняет.
9. Только после одобрения изменения появляются на сайте.

### Итог

Будущий партнерский кабинет не ломает MVP-архитектуру.

Партнер не может случайно удалить карточку, историю заявок, отзывы, SEO-страницу и аналитику.

- партнерский Telegram-бот;
- админка;
- заявки;
- статусы;
- отзывы;
- статьи;
- комиссии;
- будущий партнерский доступ.
