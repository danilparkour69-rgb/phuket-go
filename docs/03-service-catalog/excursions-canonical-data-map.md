# Canonical Excursions Data Map

Статус: [x] сводная рабочая база создана

Дата: 2026-06-29

## Главное решение

Файл `excursions-canonical-data-mvp.json` - это единая сводная рабочая база экскурсий для проверки владельцем проекта и будущего переноса в сайт/backend.

Он не удаляет и не заменяет исходники. Старые документы остаются нужны как след данных и страховка.

## Где лежит файл

- `docs/03-service-catalog/excursions-canonical-data-mvp.json`

## Из чего он собран

- краткая сайтовая таблица: `docs/03-service-catalog/excursions-site-data-mvp.json`;
- полные карточки: `docs/03-service-catalog/excursions/*.md`;
- фото и статусы фото: `docs/03-service-catalog/media/excursions/{folder}/media-manifest.json`;
- локальные фото для карусели: `docs/03-service-catalog/media/excursions/{folder}/final/carousel/`.

## Что внутри

В файле 54 экскурсии.

По каждой экскурсии собраны:

- ID, slug, город, страна, язык и категория;
- название из источника, H1 и будущий URL;
- ссылка на исходник;
- краткая фактура из раздела `Коротко`;
- цена, варианты, валюта, комиссия и ценовые пометки;
- длительность и источник расчета длительности;
- расписание и маршрут;
- что включено;
- что оплачивается отдельно;
- что взять с собой;
- важные условия и что нужно уточнить;
- кому подойдет и кому может не подойти;
- SEO title, SEO description и SEO-черновик;
- список фото из `media-manifest.json` с локальными путями и будущими публичными URL;
- подготовительный блок Tripadvisor для будущего рейтинга;
- флаги безопасности и внутренние статусы.

## Tripadvisor

Для будущего рейтинга Tripadvisor подготовлены:

- блок `tripadvisor` внутри каждой экскурсии в `excursions-canonical-data-mvp.json`;
- файл сопоставления `docs/03-service-catalog/tripadvisor-location-matching-mvp.json`;
- инструкция по интеграции `docs/03-service-catalog/tripadvisor-rating-integration.md`.

Сейчас `location_id` не заполнены, рейтинг не синхронизирован и `display_allowed` стоит `false`. Это специально: сначала нужно получить официальный API key, найти и вручную подтвердить объекты Tripadvisor, а потом подключать backend-синхронизацию.

## Как использовать

Для продуктовой проверки смотреть этот файл как главную сводную базу.

Для исправления фактуры сначала править исходную карточку экскурсии или медиаманифест, затем обновлять сводный JSON. Так сохраняется понятный след: откуда взялась каждая цифра, фраза и фотография.

## Что не делать

- Не удалять `excursions-site-data-mvp.json`.
- Не удалять markdown-карточки в `docs/03-service-catalog/excursions/`.
- Не удалять `media-manifest.json`.
- Не считать этот файл production-базой.
- Не подключать его к backend без отдельной проверки схемы и seed-скрипта.
- Не показывать Tripadvisor-рейтинг без ручного сопоставления `location_id`, backend-синхронизации и проверки правил отображения Tripadvisor.

## Связь с будущей Postgres-базой

В backend уже есть модели `Excursion` и `ExcursionPhoto`, куда позже можно перенести данные из сводного файла.

Правильный порядок:

1. Проверить `excursions-canonical-data-mvp.json` глазами.
2. Исправить спорные факты в исходных карточках или медиаманифестах.
3. Обновить сводный JSON.
4. После команды на разработку адаптировать seed/import под поля `route`, `duration`, `included`, `notIncluded`, `takeWithYou`, `restrictions` и `photos`.

## Текущие ограничения

- Фото технически собраны, но перед публикацией нужна ручная визуальная проверка.
- Условия, ограничения и доплаты остаются осторожными, если партнер не подтвердил детали.
- Для некоторых шоу и арендных форматов длительность зависит от выбранного сеанса или программы.
- Публичные тексты можно улучшать отдельно, не теряя исходную фактуру.
