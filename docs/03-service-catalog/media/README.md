# Media

Статус: [~] локальное хранение фото на этапе подготовки

## Решение

Сначала фотографии храним локально в проектной папке.

Постоянное хранилище выберем позже, перед публикацией или на этапе разработки сайта.

## Папки

- `excursions/` - фотографии экскурсий.

## Правила

Для каждой экскурсии создаем отдельную папку по ID и slug.

Пример:

- `excursions/018-phi-phi/`

Внутри актуальной рабочей структуры:

- `original/` - исходные фото по направлению;
- `carousel-candidates/` - кандидаты для карусели;
- `final/carousel/` - текущий набор фото для верхней карусели;
- `final/content/` - будущие фото по текстовым блокам;
- `media-manifest.json` - источник, статус, тип и путь каждого фото;
- `photo-brief.md` - будущий фотосценарий конкретной экскурсии.

`photo-brief.md` заполняем позже, когда будем финально подбирать фото руками. Сейчас правило фотосценария зафиксировано в `excursions/photo-scenario-rules.md`.

Финальная структура папок описана здесь:

- `excursions/final-media-folder-structure-step-51-result.md`

Старый минимальный слой фото удален из актуального процесса. Рабочей базой считаются папки направлений и `media-manifest.json`.

В финале у экскурсии должны быть:

- `original/` - исходники;
- `final/carousel/` - верхняя карусель из всех подходящих фото; сначала горизонтальные хорошего качества, если их нет - временно доступные фото с пометкой на замену;
- `final/content/` - фото по текстовым блокам;
- `media-manifest.json` - список фото для автоматической подгрузки.

## Текущее решение

На этапе подготовки MVP больше не зависаем на идеальном подборе фото.

Берем доступные фото, которые уже лежат в папке конкретного направления и соответствуют карточке экскурсии.

Финальную красоту, замену слабых фото, проверку лиц и ручную чистку переносим на этап перед публикацией.

Карта передачи фото в будущую разработку:

- `excursions/media-developer-handoff-map.md`

## Дубли фотографий

Сейчас на этапе подготовки одна и та же фотография может встречаться в разных направлениях, потому что мы собираем визуальную базу по экскурсиям.

Перед публикацией нужно заложить модель без лишнего дублирования:

- физически одно изображение хранится в одном месте;
- у изображения есть единый `asset_id`;
- разные страницы экскурсий ссылаются на этот `asset_id`;
- одна фотография может использоваться в нескольких карточках, если она подходит по смыслу;
- при замене или улучшении фото обновляется один файл, а не копии в разных папках.

Это нужно, чтобы не перегружать сервер одинаковыми файлами и не усложнять поддержку картинок.

Пример будущей логики:

- файл хранится как `media/assets/phuket-sea-001.webp`;
- `018-phi-phi` использует его как фото маршрута;
- `019-phi-phi-premium` использует тот же файл как фото карусели;
- в манифестах экскурсий хранится ссылка на общий файл, а не отдельная копия.

## Типы изображений

В `media-manifest.json` нужно указывать тип:

- `real` - реальное фото из источника или от партнера;
- `ai_enhanced` - реальное фото, улучшенное нейросетью;
- `ai_generated` - изображение создано нейросетью с нуля.

## Правило по AI

AI-изображения можно делать максимально реалистичными и красивыми, но нельзя выдавать их за реальное фото конкретной экскурсии.

Для карточек экскурсий реальные фото имеют приоритет.

AI-изображения можно использовать:

- для статей;
- для рекламных материалов;
- для эмоциональных обложек;
- как временную иллюстрацию, если реального фото пока нет.

Если AI-изображение показывает конкретное место, лодку, маршрут, отель, гида или услугу, нужно отдельно проверить, не создает ли оно ложное ожидание.

## Правило параллельной генерации

Если нужно генерировать AI-фото, эту задачу нужно запускать параллельно через субагента, чтобы основная работа по структуре, текстам и проверке не останавливалась.

Промпт для генерации должен строиться от контекста конкретной экскурсии:

- название экскурсии;
- публичный текст страницы;
- эмоция, которую продаем;
- реальные ограничения маршрута;
- запрет на ложные ожидания;
- запрет выдавать AI-фото за реальное фото.

Phuket Go продает не экскурсию как товар, а мечту, воспоминание и ощущение. Поэтому AI-фото должно поддерживать текст карточки: человек должен представить, что он уже там побывал, но без выдуманных деталей конкретной услуги.

## Что нельзя делать

- Не хранить фото только в Telegram или Telegra.ph без локальной копии.
- Не использовать внешнее хранилище как обязательное решение до разработки.
- Не выводить на сайт технические ссылки вида `/file/...`.
- Не выдавать AI-изображение за реальное фото экскурсии.
- Не подбирать галерею как случайный набор красивых фото без связи с текстом.
