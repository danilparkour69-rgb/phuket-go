# Media Manifest Site Loading Rules

Статус: [x] правило подгрузки фото для будущего сайта описано

Дата: 2026-06-28

## Зачем нужен файл

Этот документ объясняет, как будущий сайт Phuket Go должен брать фотографии из папок экскурсии.

Код проекта пока не пишем.

## Главное правило

У каждой экскурсии есть свой файл:

```text
media-manifest.json
```

В нем перечислены фотографии экскурсии, их роль и статус.

Сайт не должен вручную угадывать, какие фото показывать. Он должен читать список из `media-manifest.json`.

## Как сайт берет карусель

Для верхней карусели страницы экскурсии сайт берет все фото, где:

- `role`: `carousel`;
- файл находится в `final/carousel`;
- `manual_review` не запрещает показ.

Сайт показывает все подходящие фото, которые отобраны для карусели в `media-manifest.json`.

## Как сайт относится к статусам фото

| Статус | Что значит | Можно ли временно показывать |
| --- | --- | --- |
| `ready_for_manual_review` | Фото технически подходит, но еще не проверено руками | Да, на локальной версии и черновой сборке |
| `needs_replacement_later` | Фото взято временно, потому что лучшего пока нет | Да, только как временный вариант |
| `approved` | Фото проверено и подходит | Да |
| `rejected` | Фото отклонено руками | Нет |
| `ai_replacement_needed` | Нужно сгенерировать или улучшить фото | Нет для финальной публикации |

Перед публикацией на сайт у фото должен быть статус `approved`.

## Что показываем сейчас

Сейчас все фото считаются временными.

Текущая задача - не сделать финальный идеальный набор, а не оставить направления без визуала.

Поэтому:

- если есть хорошие горизонтальные фото, берем их;
- если хороших горизонтальных нет, берем то, что есть;
- слабые фото помечаем как `needs_replacement_later`;
- ручную чистку пользователь сделает позже.
- на этапе подготовки MVP больше не зависаем на фото: главное, чтобы фото было доступно в папке направления и соответствовало карточке экскурсии.

## Как сайт берет фото внутри текста

Фото внутри текста будущей страницы берутся из массива `content_photos`.

Пока `content_photos` пустой, потому что мы еще не делали финальную ручную раскладку фото по смыслу текста.

Позже для каждого фото внутри текста нужно будет указать:

- к какому блоку текста оно относится;
- что на нем должно быть;
- alt-текст;
- порядок показа.

## Что нельзя делать

- Нельзя брать случайные фото из всей папки экскурсии.
- Нельзя брать фото из `original` напрямую на сайт.
- Нельзя брать фото из отзывов.
- Нельзя удалять фото автоматически без ручного решения.

## Будущая оптимизация

Позже одинаковые фото не должны физически храниться много раз.

Будущая модель:

- одно фото хранится один раз;
- у него есть `asset_id`;
- разные экскурсии могут ссылаться на один `asset_id`;
- при замене фото обновляется один файл, а не много копий.

Сейчас это только архитектурная заметка. На текущем этапе фото лежат внутри папок направлений.
