# Google Sheets Final Map

Статус: [x] done

Кодинг проекта пока не начинаем.

Этот документ описывает финальную структуру Google Sheets для запуска MVP Phuket Go.

## Главное правило

Google Sheets в MVP - временное хранилище заявок.

Одна строка = одна заявка.

Google Sheets не является полноценной базой всего проекта.

Позже данные переносятся в Postgres.

## Листы

В MVP используем один основной лист:

- `Заявки`

Не создаем отдельные листы:

- `Услуги`;
- `Партнеры`;
- `Комиссии`;
- `Пользователи`.

Эти данные должны жить в будущей базе, а не разрастаться в Google Sheets.

## Допустимые статусы заявки

| Статус | Что означает | Кто ставит |
| --- | --- | --- |
| `new` | Новая заявка | Система |
| `waiting_partner` | Ожидаем ответ партнера | Система или админ |
| `accepted` | Взята в работу | Партнер или админ |
| `declined` | Отклонена | Партнер или админ |
| `completed` | Услуга оказана | Партнер или админ |
| `cancelled` | Заявка отменена | Админ |

## Допустимые статусы комиссии

| Статус | Что означает | Кто ставит |
| --- | --- | --- |
| `not_accrued` | Комиссия еще не начислена | Система |
| `accrued` | Комиссия начислена | Система или админ |
| `paid` | Партнер оплатил | Админ |
| `debt` | Есть долг партнера | Админ |

## Обязательные поля при создании заявки

Минимально нужны:

- `customer_name`;
- `customer_phone`;
- `excursion_id`;
- `excursion_title_snapshot`;
- `partner_id`;
- `status`;
- `created_at`.

Если нет этих данных, заявка не должна считаться корректной.

## Кто какие поля меняет

### Пользователь

Пользователь заполняет только форму:

- имя;
- телефон;
- Telegram;
- дата;
- количество людей;
- комментарий.

### Система

Система заполняет:

- ID заявки;
- номер заявки;
- дату создания;
- источник;
- экскурсию;
- партнера;
- цену;
- комиссию;
- статус;
- отправку Telegram-уведомлений;
- технические поля миграции.

### Партнер

Партнер через Telegram-бот может менять только:

- `status`;
- `accepted_at`;
- `declined_at`;
- `completed_at`;
- `partner_note`.

Партнер не редактирует:

- данные клиента;
- цену;
- комиссию;
- экскурсию;
- партнера;
- заметку админа.

### Админ

Админ может менять:

- статус заявки;
- партнера;
- заметку админа;
- статус комиссии;
- период комиссии;
- отмену заявки;
- пометку долга.

## Расчет комиссии

Для экскурсий MVP:

`commission_total_thb = commission_per_person_thb * people_count`

Базовое значение:

`commission_per_person_thb = 100`

Если `people_count` не заполнен, итоговую комиссию нельзя считать финальной.

Комиссия начисляется после статуса:

`completed`

## Правила цены

Публичная цена для пользователя:

- `price_rub_snapshot`

Основная цена партнера и публичная крупная цена:

- `price_thb_snapshot`

Для русской версии дополнительно сохраняем:

- `price_rub_snapshot`;
- `rub_rate_snapshot`;
- `rate_date`;
- `currency_note_shown`.

Текст под ценой:

`примерно {price_rub} ₽ по текущему курсу`

## Что не добавляем в Google Sheets MVP

Не добавляем:

- язык гида;
- район отеля клиента;
- трансфер по району;
- рекламные метки;
- отдельную таблицу партнеров;
- отдельную таблицу экскурсий;
- отдельную таблицу пользователей;
- отдельную таблицу комиссий.

Причина: Google Sheets нужен только как временная таблица заявок.

## Что важно для переноса в Postgres

Каждая строка Google Sheets должна быть переносима в будущие сущности:

- `Lead`;
- `LeadStatusHistory`;
- `Commission`;
- связь с `User`, если есть `user_id`;
- связь с `Partner`;
- связь с `Excursion`.

Поле `migration_status` нужно, чтобы понимать:

- заявка еще не перенесена;
- заявка успешно перенесена;
- при переносе была ошибка.

## Definition of Done этапа

Этап Google Sheets считается готовым, если:

- есть финальный список колонок;
- понятно, какие поля обязательны;
- понятно, кто какие поля заполняет;
- описаны статусы заявки;
- описаны статусы комиссии;
- описан расчет комиссии;
- учтен перенос в Postgres;
- Google Sheets не разрастается в полноценную базу.
