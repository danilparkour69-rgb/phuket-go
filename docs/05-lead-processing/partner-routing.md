# Partner Routing

Статус: [~] draft

## Правила маршрутизации

В MVP все заявки обязательно уходят владельцу проекта в Telegram-бот.

Дополнительно заявка должна уходить исполнителю/партнеру в Telegram-бот. Маршрутизация может зависеть от карточки услуги, категории, партнера или ручной настройки администратора.

## Специальное правило для дайвинга

В дальнейшем дайвинг отправляется отдельному исполнителю.

Правило:

- все обычные экскурсии уходят основному исполнителю экскурсий;
- заявки по дайвингу уходят отдельному исполнителю дайвинга;
- владелец проекта все равно получает копию каждой заявки;
- в админке должна быть возможность вручную переназначить исполнителя, если маршрут нужно изменить.

Это правило нужно заложить в архитектуру маршрутизации, даже если в первой локальной версии исполнитель задается вручную.

Подробная проверка дайвинга сохранена в:

- `../03-service-catalog/excursions/diving-routing-step-43-result.md`

Для пользователя это не должно выглядеть как отдельный партнер. Пользователь видит обычную карточку Phuket Go и оставляет заявку. Разделение исполнителей происходит внутри админки и Telegram-бота.

Минимальное правило для будущей логики:

- если карточка имеет ID `032` или служебный тип `diving`, заявка идет исполнителю дайвинга;
- если карточка не имеет типа `diving`, заявка идет основному исполнителю экскурсий;
- админ получает копию всегда.

## Резервный сценарий

Если партнер не отвечает или маршрут не настроен:

- заявка остается у владельца проекта/администратора;
- пользователю отправляется сообщение "Ожидаем подтверждения";
- заявка не считается срочной.

## Дублирование владельцу платформы

Дублирование владельцу платформы обязательно для всех заявок на MVP.

## Время ответа

Жесткого времени ответа партнера в MVP нет.

Заявка считается несрочной.
