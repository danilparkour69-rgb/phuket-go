# PROJECT_INSTRUCTIONS_FOR_CODEX.md

Эта инструкция предназначена для Codex. Следуй ей при разработке проекта.

## 1. Контекст проекта

Проект состоит из нескольких частей:

```text
backend/              # API, авторизация, бизнес-логика, интеграции
web/ или frontend/    # закрытое веб-приложение после логина
landing/ или website/ # публичный лендинг/сайт до регистрации
mobile/               # мобильное приложение на React Native/Expo
telegram-mini-app/    # мини-приложение в Telegram
docs/                 # requirements.md, tasks.md, архитектурные решения
infra/                # деплой, Docker, окружения, если нужно
```

Не превращай проект в один монолитный frontend. Web, landing, mobile и Telegram Mini App — разные клиенты. Backend/API и база — общая основа.

## 2. Главные правила

Источник правды — текущий код, схемы БД, тесты и актуальные документы в `docs/`. Документация может устареть, поэтому при противоречии сначала разберись в коде и предложи выравнивание.

Пользователь не программист. Не спрашивай его, какой enum, индекс, ORM-паттерн, middleware или SQL-схему выбрать. Переводи технические развилки на продуктовый язык: кто что видит, кто что может сделать, какие данные приватные, какие сценарии входят в MVP.

Используй существующий стек. Не заменяй стек без явной причины и плана.

Не подключай платные сервисы, платные тарифы, облачные ресурсы, платные API, платёжные провайдеры, платные сборки, аккаунты магазинов приложений или платные UI-компоненты без явного решения пользователя. Если не уверен, бесплатен ли инструмент сейчас, считай его потенциально платным и спроси пользователя.

Не добавляй production-зависимости просто потому, что так удобнее. Сначала проверь, есть ли решение в проекте. Если новая зависимость действительно нужна, объясни, какую проблему она решает и почему существующих средств недостаточно.

Не убивай процессы и не освобождай порты без необходимости. Если порт занят, сначала выясни, кто его использует и относится ли процесс к текущему проекту.

Не храни секреты в коде, документации, логах и коммитах. Реальные `.env`, `.env.production`, ключи, токены, JWT_SECRET, storage keys и API keys не должны попадать в git.

## 3. Стек по умолчанию

Используй этот стек, пока пользователь не принял другое решение:

- TypeScript;
- React + Vite для `web/frontend`;
- TanStack Query, TanStack Router, TanStack Form;
- shadcn/ui как UI-база;
- Bun + Hono для `backend`;
- Prisma + Zod + PostgreSQL;
- Playwright для web-e2e;
- React Native + Expo для `mobile`;
- Maestro для mobile-e2e;
- Docker для локальной/test PostgreSQL, если это упрощает воспроизводимость;
- S3-compatible storage только когда нужны файлы;
- Redis/Valkey только когда нужны Pub/Sub, cache или общий rate limit;
- DigitalOcean или Yandex Cloud как развилка для пользователя, не как автоматический выбор.

Рендеринг:

- `web/frontend` — CSR, если это закрытый кабинет за логином;
- `landing/website` — SSG/static;
- открытый каталог/SEO-страницы — отдельная развилка, спроси пользователя;
- `mobile` — React Native/Expo, не WebView;
- `telegram-mini-app` — web-клиент внутри Telegram, но с серверной проверкой Telegram initData.

## 4. Правила работы по задачам

Не начинай реализацию с кода. Для нетривиальных задач сначала дай план.

Обычный цикл:

1. Прочитай `AGENTS.md`, `docs/requirements.md`, `tasks.md` и релевантный код.
2. Найди первую незакрытую задачу или уточни задачу пользователя.
3. Опиши цель, риски, зависимости, edge cases и критерии готовности.
4. Если задача меняет поведение — работай TDD-first.
5. Реализуй минимально достаточное решение.
6. Запусти релевантные тесты, типы, сборку или линт.
7. Запусти независимое code review через сабагента или отдельный review pass.
8. Исправь обоснованные замечания.
9. Повтори review до 9.5/10 или 10/10 без требований к доработке.
10. Сделай осмысленный commit, если изменения завершены и проверены.
11. Отметь задачу в `tasks.md`.

Останавливайся и спрашивай пользователя, если:

- нужна продуктовая развилка;
- действие платное или потенциально платное;
- меняется архитектура;
- затрагиваются деньги, платежи, безопасность, приватность или юридические требования;
- нужно выбрать cloud provider, регион, тариф, платёжного провайдера, магазинный аккаунт;
- решение может удалить данные или испортить production.

## 5. Режимы работы

### Direct

Используй для простых правок: текст, мелкое переименование, очевидная косметика. Не устраивай большое расследование ради одного слова.

### Investigation

Используй для багов. Найди корень, а не симптом. Если баг виден в компоненте, проверь родителя, API, состояние, маршрутизацию, данные и тесты.

### TDD-first

Используй для поведения:

- авторизация;
- роли и права;
- API-контракты;
- сохранение данных;
- валидация;
- платежи;
- расчёты;
- статусы;
- Telegram initData verification;
- mobile critical flows;
- security-sensitive fixes.

Цикл: один падающий тест → минимальная реализация → зелёный тест → следующий случай.

Не применяй TDD механически к каждой косметической правке.

### Review-only

Если пользователь просит review, не исправляй сразу. Сначала дай список проблем с `file:line`, риском и обоснованием. Исправления — только после отдельной команды или если review prompt явно просит исправлять.

## 6. Требования и tasks.md

`docs/requirements.md` должен быть чистым документом с выбранными решениями. Не оставляй там историю обсуждения, варианты, черновики и «мы выбирали между A и B».

`tasks.md` должен быть рабочей очередью. Каждая задача должна иметь:

- цель;
- источник из requirements;
- объём;
- роли/права, если применимо;
- критерий готовности;
- тесты;
- статус.

Не делай задачи параллельно, если они конфликтуют по одной и той же схеме данных, ролям, авторизации, API или состоянию.

## 7. Тесты

Тесты должны защищать поведение, а не случайную старую вёрстку.

Используй:

- backend unit/integration tests для логики, API, auth, ролей, валидации;
- Playwright для web-e2e;
- Maestro для mobile-e2e;
- отдельные smoke/e2e для Telegram Mini App, когда он появится;
- test database через `TEST_DATABASE_URL`;
- seed/demo data для повторяемых сценариев.

Плохой тест: проверяет только текст, который легко изменится, и не ловит поведение.

Хороший тест: ловит критичный сценарий, право доступа, ошибку, сохранение данных или важную ветку UX.

После настройки тестов оцени доверие к ним от 1 до 10 и объясни, какие поломки они не заметят.

## 8. Git

Перед новой большой задачей проверь `git status`.

Коммить маленькими осмысленными кусками:

- после настройки каркаса;
- после ТЗ;
- после tasks.md;
- после завершённой задачи;
- после системного UI-рефакторинга;
- после security fix;
- перед рискованным экспериментом.

Не коммить незавершённые или чужие изменения. Если есть сомнения, объясни состояние и спроси.

Не настраивай CI/CD и автодеплой без явной просьбы. Для маленькой команды по умолчанию безопаснее ручной деплой.

## 9. UI-система

Используй shadcn/ui и общие компоненты системно.

Правила:

- косметика живёт внутри компонента;
- layout живёт в родителе или обёртке;
- снаружи не прокидывай `style`/`className` для одноразовой косметики;
- компонент принимает осмысленные props: `variant`, `size`, `tone`, `disabled`;
- повторяющиеся таблицы, фильтры, карточки, бейджи, чекбоксы, меню, пустые состояния и skeleton-loader'ы выноси в общие компоненты;
- не чини auth/layout баги косметикой.

Если на защищённой странице мелькает публичное состояние, исследуй auth guard, routing и loading state. Не прячь баг задержкой или CSS.

## 10. Backend, база и авторизация

Frontend, mobile и Telegram Mini App не ходят напрямую в базу. Только через backend/API.

Backend отвечает за:

- авторизацию;
- роли;
- права доступа;
- проверку владельца объекта;
- бизнес-инварианты;
- расчёты;
- статусы;
- платежные статусы;
- storage permissions;
- Telegram initData verification.

База:

- PostgreSQL;
- Prisma migrations;
- локальная dev DB и отдельная test DB;
- индексы добавлять под реальные запросы;
- unique constraints использовать для инвариантов вроде уникального email;
- seed не должен тащить опасные production-дефолты.

Авторизация:

- access token короткоживущий;
- refresh token хранить безопасно, желательно hash в базе;
- cookie: HttpOnly, Secure в production, SameSite;
- Origin/CSRF checks там, где cookie участвует в действии;
- не возвращать refresh token в JSON, если выбран HttpOnly-cookie подход;
- logout должен отзывать/закрывать сессию корректно.

## 11. Telegram Mini App

Telegram Mini App — отдельный frontend-клиент внутри Telegram WebView.

Обязательные правила:

- отдельная папка `telegram-mini-app`;
- общий backend/API;
- Bot Token только на backend;
- клиент отправляет `window.Telegram.WebApp.initData` на backend;
- backend проверяет подпись initData по правилам Telegram;
- `initDataUnsafe` не доверять;
- Telegram user ID связывать с пользователем проекта на backend;
- права доступа проверять на backend;
- не переносить админку в Telegram Mini App без решения пользователя;
- Telegram Payments или другие платные/юридические сценарии — только после решения пользователя.

## 12. Mobile

Mobile — React Native/Expo, не WebView.

Правила:

- используй общий backend/API;
- не дублируй бизнес-логику;
- сначала реализуй клиентские сценарии, если админка удобнее в web;
- проверяй через simulator и реальные устройства;
- используй Maestro для критичных сценариев;
- EAS paid plans, Apple Developer Program, Google Play Console, TestFlight/Play release — только после решения пользователя;
- in-app purchases и подписки — отдельный план, отдельные тесты, отдельное ревью.

## 13. Deploy

По умолчанию deployment decision остаётся за пользователем.

Возможная схема:

- backend/API — container или web service;
- web/frontend CSR — static site/CDN;
- landing SSG — static site/CDN;
- database — managed PostgreSQL, если пользователь согласовал платный ресурс;
- storage — S3-compatible, если нужны файлы;
- worker — если есть долгие задачи;
- cron — если есть задачи по расписанию;
- Redis/Valkey — если нужны Pub/Sub, cache, общий rate limit;
- autodeploy off, пока пользователь не решил иначе.

Перед созданием ресурсов дай список платных действий и попроси подтверждение.

## 14. Production checks

Перед production проверь:

- env variables;
- CORS/origins;
- cookie flags;
- database migrations;
- seed не запускается опасно;
- health-check;
- logs;
- error monitoring plan;
- key user flows;
- roles/menu;
- logout/redirect;
- loading states;
- forms validation;
- storage permissions;
- payment stubs отсутствуют;
- secrets не в git;
- security audit выполнен.

## 15. Security

Особо проверяй:

- IDOR: пользователь не может читать/менять чужие объекты по ID;
- роли и права;
- JWT_SECRET и другие секреты;
- refresh token flow;
- CSRF/Origin;
- CORS;
- upload/storage;
- external image URLs;
- payment flows;
- dependency warnings;
- Docker hardening;
- logs без секретов;
- rate limit;
- Telegram initData verification.

Для каждой найденной проблемы укажи риск, сценарий, критичность, минимальный фикс и тест.

## 16. Готовые промпты

### Сбор requirements.md

```text
Я не программист, техническая основа уже готова в проекте — смотри стек по репозиторию. Мне нужно отвечать на вопросы только с продуктовой точки зрения.

Сначала задай все важные продуктовые вопросы одним списком. К каждому вопросу дай 2-3 варианта ответа и отметь рекомендуемый. Не задавай вопросы по одному — собери всё сразу.

После моих ответов собери финальный `docs/requirements.md`: только выбранные решения, без следов обсуждения и без вариантов ответа.

Если вопрос технический, переведи его на продуктовый язык и предложи варианты. Платные сервисы, тарифы и провайдеры не выбирай без моего решения.
```

### Слепое независимое code review

```text
Создай сабагента для слепого независимого code review активных изменений.

Пусть он:
1. посмотрит изменения без контекста нашего диалога;
2. оценит результат от 1 до 10;
3. перечислит только реальные проблемы: баги, риски, поломанные сценарии, слабые тесты;
4. спорные замечания отдельно вынесет на обсуждение.

Исправь обоснованные замечания. Потом создай нового сабагента и повтори цикл, пока оценка не будет 10/10 или хотя бы 9,5/10 без требований к доработке. Если оценка ниже 9, продолжай ревью и исправления.
```

### Автономный цикл по tasks.md

```text
Посмотри `tasks.md`, найди первую незакрытую задачу.

Перед реализацией запусти сабагента, который опишет подводные камни и важные краевые случаи.

Реализуй задачу. Потом запусти сабагентов на code review по одному, каждый ставит оценку от 1 до 10. Не выходи из цикла, пока оценка не будет 9.5/10 или выше: разбирай замечания, принимай только относящиеся к задаче, вноси правки и снова проверяй.

Когда задача принята — сделай commit, отметь задачу выполненной в `tasks.md` и переходи к следующей. Продолжай, пока все задачи не закрыты или пока не упрёшься в решение, которое должен принять пользователь.

Платные сервисы, тарифы, облака и платёжные провайдеры не выбирай без пользователя.
```

### Оценка AGENTS.md

```text
Изучи `AGENTS.md`. Дай ему оценку от 1 до 10 с разных точек зрения, включая лучшие практики промптинга.

Покажи, где инструкция перегружена, где нет приоритетов, где правила проверки результата слишком тяжёлые для маленьких задач, где не хватает явного правила про зависимости, платные сервисы или режим code review.

В конце предложи конкретные улучшения. Не меняй файл без отдельного подтверждения.
```

### Системный UI-рефакторинг

```text
Проведи системный рефакторинг UI всего фронтенда.

Правила:
- Вся косметика живёт строго внутри компонента.
- Снаружи компонент принимает только понятный API: variant, size, tone, disabled и другие осмысленные параметры.
- За расположение на странице отвечает родительский layout или специальная обёртка.
- Повторяющиеся элементы выносим в общие компоненты и приводим к единому виду.

Найди места, где система расползается, и выровняй их. На выходе: меньше одноразовых классов, меньше хардкода, ясная граница между «как выглядит» и «где стоит».
```

### Системная валидация форм

```text
Сделай нормальную валидацию формы: не блокируй кнопку молча, а показывай понятные ошибки. Сначала общая ошибка под формой, затем ошибки рядом с конкретными полями.

Когда механика готова на одной форме — пройди системно по всем остальным формам проекта и реализуй точно такую же валидацию. Если по дороге чинишь баг — сразу напиши на него тест.
```

### Security audit

```text
Проведи security audit перед production.

Проверь JWT_SECRET, access/refresh token flow, cookie flags, CSRF/Origin checks, отсутствие секретов в git, Dockerfile, зависимости, storage, внешние URL, платежи, IDOR, роли и права, mobile на реальных устройствах, telegram-mini-app initData verification на backend.

Для каждой проблемы укажи риск, сценарий эксплуатации или поломки, критичность, минимальный фикс и тест. Не делай вид, что аудит гарантирует абсолютную безопасность: нужны итерации и независимое ревью.
```
