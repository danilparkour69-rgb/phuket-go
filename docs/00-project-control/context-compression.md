## Ключевые решения Phuket Go

- MVP только по Пхукету.
- Аудитория MVP: русскоязычные туристы.
- MVP-категория: экскурсии.
- Каждая экскурсия получает отдельную SEO-страницу.
- Оплаты на сайте нет.
- Phuket Go не продает экскурсии напрямую и не принимает оплату на сайте.
- Модель денег: обратная комиссия от партнера после оказания услуги.
- Комиссия по экскурсиям: 100 THB за человека.
- Публичная цена MVP показывается в батах крупно.
- Для русской версии рядом мелко показывается примерная цена в рублях.
- В базе, для партнера и оплаты основная цена хранится в THB.
- Временное хранение заявок: Google Sheets.
- Целевая база: Postgres.
- Первый запуск: локально на компьютере.
- Публикация позже: только после локального 10/10 и отдельного решения владельца проекта; provider/domain details брать из актуального решения и deployment docs.
- Авторизация MVP: Telegram, Google, email. Телефон через SMS не используем.
- Партнерский кабинет в MVP не нужен.
- Партнер работает через Telegram-бот.
- Все обычные экскурсии идут основному исполнителю.
- Дайвинг идет отдельному исполнителю.
- Владелец проекта получает копию всех заявок.
- Админка нужна в MVP.
- Будущий партнерский доступ закладываем архитектурно, но не реализуем в MVP.
- В будущем партнер не сможет удалить карточку, только скрыть или отправить на модерацию.
- Если решения шаблона `vibe` конфликтуют с ранними техническими решениями Phuket Go, за основу берем `vibe`.
- Если сохраненная инструкция Codex конфликтует с `AGENTS.md`, README шаблона `vibe` или актуальными документами Phuket Go, приоритет такой: системные/developer-инструкции, `AGENTS.md`, актуальные `docs/`, затем `project-instructions-for-codex.md`.
- В сохраненной инструкции Codex общие названия `web/landing` адаптируются под шаблон `vibe`: публичный сайт и SEO-каталог - `website`, закрытые кабинеты - `webapp`, API - `backend`.
- Масштабирование закладываем сразу: Phuket Go -> Bangkok Go -> Pattaya Go -> Thailand Go -> Vietnam Go -> Asia Go -> World Go.

## Контент и позиционирование

- Мы продаем не экскурсии, а эмоции, мечту, воспоминания и ощущение будущей поездки.
- Тон: вежливый, дружелюбный, на "вы", без панибратства.
- Тексты должны сначала вызывать желание, потом давать информацию.
- Для статей: статья продает эмоцию или решение проблемы, а карточка услуги появляется органично в конце.
- SEO-фокус MVP: экскурсии на Пхукете и полезные статьи для туристов.

## Фото

- Фото храним локально до выбора постоянного хранилища.
- Потом возможен перенос в Cloudinary или другое хранилище.
- Сейчас используем все доступные фото направления, если они соответствуют карточке.
- Ограничения 4-5 фото для карусели больше нет.
- Для карусели предпочтительны горизонтальные фото хорошего качества.
- Если хороших фото нет, временно берем доступные и помечаем для будущей замены.
- Ручную чистку неподходящих фото пользователь сделает позже.
- Перед публикацией нужно проверить лица людей, повторы, слабые фото, фото из отзывов и соответствие карточке.
- Будущая модель хранения: одно физическое фото хранится один раз и используется через общий `asset_id`.
- AI-изображения можно использовать позже, но маркировать как `real`, `ai_enhanced`, `ai_generated`.
